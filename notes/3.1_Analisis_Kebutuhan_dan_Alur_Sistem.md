# 3.1 Analisis Kebutuhan dan Alur Sistem

Tahap analisis kebutuhan sistem merupakan langkah krusial untuk memastikan bahwa perangkat lunak yang dibangun dapat menyelesaikan permasalahan pengarsipan secara efisien. Sistem Informasi Administrative Files Archive (DAFA) dirancang untuk memfasilitasi dua aktor utama: Administrator (Admin) dan Client (Pegawai/Guru).

## 3.1.1 Analisis Kebutuhan Fungsional

Kebutuhan fungsional memuat fungsionalitas utama yang harus dapat dilakukan oleh sistem berdasarkan peran penggunanya (*Role-Based Access Control*).

**A. Kebutuhan Fungsional Administrator (Admin)**
1. Admin dapat melakukan login ke dalam sistem menggunakan kredensial yang valid.
2. Admin dapat mengelola (*Create, Read, Update, Delete*) data pengguna dan mendefinisikan matriks hak akses (*role assignment*) secara spesifik untuk masing-masing Client.
3. Admin dapat menetapkan dan mengelola Tahun Akademik atau Tahun Pelajaran yang sedang aktif.
4. Admin memiliki akses penuh untuk mengelola struktur arsip, mengunggah dokumen, menghapus, serta mengubah nama dokumen tanpa batasan direktori.

**B. Kebutuhan Fungsional Client (Pegawai)**
1. Client dapat melakukan login ke dalam sistem menggunakan kredensial yang diberikan oleh Admin.
2. Client dapat melihat hierarki folder dan arsip yang secara eksplisit diizinkan oleh Admin sesuai dengan jabatan/tupoksinya.
3. Client dapat mencari dokumen arsip tertentu berdasarkan nama file atau kategori.
4. Client dapat mengunduh dokumen arsip ke penyimpanan lokal.

## 3.1.2 Analisis Kebutuhan Non-Fungsional

Kebutuhan non-fungsional mendefinisikan standar teknis, kualitas, dan batasan infrastruktur yang menjaga stabilitas sistem.

1. **Keamanan (Security)**: Password pengguna harus dienkripsi menggunakan metode *hashing* (bcrypt) sebelum disimpan ke dalam basis data. Proses pertukaran data antar sisi klien dan peladen menggunakan token otorisasi (JSON Web Token/JWT) yang disimpan secara aman dalam *HTTP-only cookies*.
2. **Kinerja (Performance)**: Waktu respons pengunduhan arsip harus optimal. Untuk arsip berupa struktur folder (direktori), sistem harus mampu melakukan *zipping* (kompresi) secara langsung (*on-the-fly*) menggunakan pustaka *archiver* sebelum dikirimkan ke pengguna.
3. **Ketersediaan (Availability)**: Sistem dirancang menggunakan arsitektur antarmuka aplikasi terpisah (*decoupled architecture*) antara Frontend dan Backend agar dapat direplikasi atau diperbarui tanpa harus mematikan seluruh ekosistem.
4. **Antarmuka Pengguna (Usability)**: Antarmuka harus responsif (dapat diakses secara optimal baik di perangkat desktop maupun *mobile*) serta intuitif bagi pengguna awam.

## 3.1.3 Analisis Alur Kerja Sistem (Workflow)

Secara umum, alur kerja sistem diatur berdasarkan matriks akses. 
1. Saat pengguna masuk ke halaman utama (*Dashboard*), komponen frontend akan meminta data arsip ke *endpoint* API backend. 
2. Backend akan membaca identitas pengguna dari token yang dikirimkan.
3. Apabila identitas dikenali sebagai *Admin*, sistem akan mengembalikan seluruh pohon direktori (*directory tree*).
4. Apabila identitas dikenali sebagai *Client*, backend melakukan *query* validasi silang (JOIN) terhadap tabel `client_access` untuk memfilter arsip mana yang sesuai dengan status jabatan *Client* tersebut, menyembunyikan struktur lainnya, lalu mengirimkan respons yang sudah terisolasi kembali ke frontend.

## 3.1.4 Analisis Pembagian Hak Akses dan Tahun Pelajaran

Kunci utama dari isolasi data pada sistem DAFA terletak pada dua filter logika (klausul kondisi):

**A. Pembagian Akses Berbasis Jabatan (Client Access)**
Sistem mengakomodasi hierarki struktural sekolah (Kepala Sekolah, Tata Usaha, Bendahara, Waka Kurikulum, dll). Dalam praktiknya:
- Saat Admin menambahkan arsip/folder baru, Admin wajib memberikan *tag* kepemilikan (misal: arsip ini milik divisi "Kurikulum").
- Tabel `client_access` di basis data menyimpan nilai *boolean* (Benar/Salah) untuk setiap posisi jabatan bagi tiap pengguna.
- Seorang *Client* hanya akan bisa melihat arsip apabila ia memiliki nilai TRUE pada jabatan yang cocok dengan *tag* arsip tersebut.
- Hal ini memungkinkan satu entitas *Client* memiliki **jabatan rangkap** (contoh: seseorang menjabat sebagai "Guru Piket" sekaligus "Wali Kelas"), sehingga sistem secara dinamis menggabungkan (UNION) tampilan arsip dari kedua jabatan tersebut pada dasbor miliknya.

**B. Pembagian Akses Berbasis Tahun Pelajaran (Tapel)**
- Untuk mencegah penumpukan dokumen dari tahun ke tahun yang dapat memicu kebingungan, sistem memisahkan arsip secara mutlak berdasarkan **Tahun Pelajaran (Tapel)**.
- Admin mengatur satu "Tahun Pelajaran Aktif" pada *global settings*.
- Semua aktivitas *query* dari *Client* akan secara otomatis difilter hanya untuk menampilkan arsip yang diunggah pada Tahun Pelajaran yang sedang aktif saat ini. Arsip dari tahun-tahun sebelumnya tetap tersimpan aman di direktori server dan basis data, namun tersembunyi dari dasbor aktif *Client* hingga Admin mengubah kembali status Tapel tersebut.
